🚀 Mamba-2 推理生成流程

训练好的模型如何一个字一个字地生成文本？Mamba 的推理为什么比 Transformer 高效？

推理 vs 训练：核心区别

训练时，我们一次喂入完整序列 \([B, L, d_{\text{model}}]\)，并行处理所有 token。

推理时，我们一次只生成一个 token，然后把它拼到序列末尾，再生成下一个——自回归。

输入: "从前有一只" 第1步: 模型预测 → "小" → "从前有一只小" 第2步: 模型预测 → "猫" → "从前有一只小猫" 第3步: 模型预测 → "，" → "从前有一只小猫，" ...重复直到生成 <eos> 或达到最大长度

⚡ Mamba 推理的杀手锏：O(1) 内存/步 MAMBA 优势

这是 Mamba 相比 Transformer 最大的推理优势。

Transformer 推理

每生成一个 token，需要和所有之前的 token 做 Attention。

$$\text{复杂度: } O(L) \text{ 每步}$$

需要 KV Cache 存储所有历史 token 的 Key/Value，内存随序列长度线性增长。

序列长度 1000 → KV Cache ≈ 数百MB 序列长度 100000 → KV Cache ≈ 数十GB 💥

Mamba 推理

SSM 是递推结构：隐状态 \(\mathbf{h}\) 已经压缩了所有历史信息。

$$\text{复杂度: } O(1) \text{ 每步}$$

只需保存一个固定大小的隐状态，不管序列多长，内存恒定。

序列长度 1000 → 隐状态 ≈ 固定大小 序列长度 100000 → 隐状态 ≈ 同样大小 ✓

第1步: 初始化隐状态

推理开始前，每一层的 SSM 隐状态 \(\mathbf{h}\) 和 Conv1D 状态初始化为零：

$$\mathbf{h}^{(l)} = \mathbf{0} \in \mathbb{R}^{n_{\text{heads}} \times \text{headdim} \times d_{\text{state}}} = \mathbb{R}^{32 \times 64 \times 64}$$
conv_state: [32层 × d_inner × d_conv] 存最近4个token ssm_state: [32层 × 32heads × 64 × 64] SSM隐状态

这两个状态会在生成过程中逐步更新，替代 Transformer 的 KV Cache。

第2步: Prompt 编码（可选预填充）

如果用户给了一段开头（如"从前有一只"），先把这段 prompt 喂进模型更新隐状态：

for token in prompt_tokens: logits, states = model.step(token, states) # 不需要 logits, 只要更新 states

这一步可以用训练时的并行模式加速——一次处理整个 prompt。

第3步: 逐 token 生成（自回归循环）

核心循环：每步只处理一个 token。

for step in range(max_new_tokens): # 1. Embedding: token_id → 向量 x = embedding[token_id] # [1, 1, 1024] # 2. 过32层 Mamba Block (递推模式) for layer in layers: x, states[layer] = mamba_step(x, states[layer]) # 3. LM Head: 向量 → 词表概率 logits = lm_head(norm(x)) # [1, 1, 32000] # 4. 采样下一个 token token_id = sample(logits) output.append(token_id)

第4步: SSM 递推更新（每层每步）核心

每个 Mamba Block 内部，SSM 在递推模式下执行：

$$\begin{aligned} \mathbf{h}_t &= \bar{\mathbf{A}} \cdot \mathbf{h}_{t-1} + \bar{\mathbf{B}} \cdot \mathbf{x}_t \\ \mathbf{y}_t &= \mathbf{C}_t \cdot \mathbf{h}_t + \mathbf{D} \cdot \mathbf{x}_t \end{aligned}$$

只需要一次矩阵向量乘——不需要展开成矩阵模式。

训练 vs 推理：SSM 的两种模式

训练 (矩阵模式 / SSD): 所有时间步并行 → Y = M ⊙ (QKᵀ) · V 利用 Tensor Core, 吞吐量高 复杂度: O(L²) 但 GPU 并行效率极高 推理 (递推模式): 一次一步 → h(t) = Ā·h(t-1) + B̄·x(t) 只做向量运算, 不需要矩阵展开 复杂度: O(1) 每步, 内存恒定

这就是 Mamba-2 "Structured State Space Duality" (SSD) 的精髓：同一个数学公式，训练和推理用不同的计算方式。

第5步: 采样策略

LM Head 输出 32000 维的 logits 后，需要从中采样下一个 token：

$$P(y_t) = \text{softmax}\!\left(\frac{\text{logits}}{\tau}\right), \quad y_t \sim \text{TopK}(P, k)$$
参数说明: τ (temperature): τ=0.1 → 几乎确定性选最高概率 (保守) τ=1.0 → 按原始概率采样 (标准) τ=2.0 → 更随机, 更有创意 (激进) top_k=50: 只从概率最高的前50个token中采样 防止选到极低概率的"垃圾"token top_p=0.9 (nucleus sampling): 选累积概率达到90%的最小token集合 动态调整候选数量

我们的 chat.sh 使用 temperature=0.8, top_k=50。

第6步: 生成终止

循环在以下条件之一满足时终止：

1. 生成了 <eos> token (序列自然结束) 2. 达到 max_new_tokens 限制 (我们设的500) 3. 用户手动中断

最后将所有生成的 token_id 通过分词器解码回文字：

$$[1823, 945, 302, 89, 3021] \xrightarrow{\text{decode}} \text{"从前有小猫"}$$

完整推理流程回顾

$$\text{"从前有"} \xrightarrow{\text{BPE}} \text{[IDs]} \xrightarrow[\text{更新 h}]{\text{prompt编码}} \xrightarrow[\text{采样}]{\text{自回归循环}} \text{token} \xrightarrow{\text{decode}} \text{"从前有一只小猫"}$$

Mamba 推理的三大优势：

1. O(1) 内存/步 — 无需 KV Cache, 序列再长内存不增长 2. 恒定计算量/步 — 不像 Attention 随序列变慢 3. 训练/推理数学等价 — SSD 对偶性保证结果一致
